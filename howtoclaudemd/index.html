CLAUDE CODE 設定入門

CLAUDE.md はじめの一歩
— 士業・コンサルタントのための「指示書」の書き方

Claude Codeは、あなたの代わりにファイルを作り、コマンドを実行してくれるAIアシスタントです。
でも毎回ゼロから「うちのルール」を説明するのは面倒ですよね。
CLAUDE.md は、Claude Codeに渡す「うちの事務所のルールブック」です。
このガイドでは、プログラミング経験がなくても書けるように、
士業・コンサルタントの日常業務に即した言葉で解説します。

こんな方に向けて書いています
行政書士・社労士・税理士・中小企業診断士などの士業、経営コンサルタント、補助金申請支援者。
Claude Codeを使って業務を自動化したいが、エンジニア向けの説明ではピンと来ない方。

1. CLAUDE.mdって何？

一言で言えば「毎朝、助手に渡す引き継ぎメモ」です。

たとえるなら…
あなたの事務所に、毎朝「今日が初出勤」の優秀な助手が来ると想像してください。
能力は高い。でも、あなたの事務所のやり方は何も知りません。

「うちはファイル名に日付を入れるルールだよ」
「顧客への連絡はこのテンプレートを使ってね」
「この補助金の申請書はこの順番で作ってね」

毎朝これを口頭で説明するのは大変ですよね。
だから、紙に書いて渡す。これがCLAUDE.mdです。

Claude Codeは、作業を始めるたびに前回の記憶がリセットされます。昨日どんなやり取りをしたか覚えていません。でもCLAUDE.mdは、毎回自動的に最初に読んでくれます。だから「何度も同じ説明をする」がなくなります。

❌ CLAUDE.mdがない場合
「ファイル名は年月日_顧客名にして」
「Wordで作って」
「敬語で書いて」
…と毎回指示し直す
✅ CLAUDE.mdがある場合
「○○さんの申請書を作って」
の一言で、命名ルールも書式も敬語も
全部守って作ってくれる

2. どこに置く？

作業フォルダの一番上に置くだけです。

たとえばあなたの仕事用フォルダが my-project だとしたら：

フォルダの中身 
my-project/
  ├── CLAUDE.md        ← ★ ここに置く
  ├── templates/        ← 申請書テンプレートなど
  ├── clients/          ← 顧客別フォルダなど
  └── scripts/          ← 自動化スクリプトなど
⚠️ ファイル名に注意 CLAUDE.md は大文字です。claude.md（小文字）では読み込まれません。
「CLAUDE」が大文字、「.md」が小文字。これだけ守ってください。
💡 もうひとつ：CLAUDE.local.md 自分だけのメモ（テスト用の設定など）は CLAUDE.local.md に書けます。
このファイルは自動的に共有対象から外れるので、チームに見られる心配がありません。

3. 何を書く？ — 5つのブロック

全部書く必要はありません。自分の業務に合うものだけ選んでください。

  1. このプロジェクトは何か（概要）
    1〜3行でOK。「何の仕事をするフォルダか」を書く。
    例）「建設業許可申請の書類作成プロジェクト」
    例）「小規模事業者持続化補助金の事業計画書作成支援」
  2. よく使う操作の手順（コマンド）
    「コマンド」とは、Claude Codeに打ち込む操作指示のことです。
    普段やる操作を書いておくと、Claudeが正しい手順を覚えてくれます。
    例）「Pythonスクリプトを動かすとき → python scripts/generate.py」
    例）「Wordファイルをプレビューするとき → open output.docx」
  3. フォルダの役割（構造）
    「どのフォルダに何が入っているか」を書く。
    Claudeが「このファイルはどこに保存すればいいか」を迷わなくなります。
  4. 仕事のルール（決まりごと）
    命名規則、文体、フォーマットなど、事務所の「お作法」を書く。
    例）「ファイル名は 年月日_顧客名_書類名.docx」
    例）「敬体（です・ます調）で統一」
  5. やってはいけないこと（注意事項）
    Claude Codeが過去にやらかしたことや、絶対に守ってほしい禁止事項。
    例）「顧客の個人情報をファイル名に入れない」
    例）「テンプレートの元ファイルを直接編集しない。必ずコピーしてから使う」
書く基準はひとつだけ 「これを書いておかないと、Claudeが間違えること」だけ書く。
当たり前のこと（「日本語で書いてね」など）は書かなくて大丈夫です。

4. 実例 ① 行政書士事務所の場合

建設業許可申請の書類作成を支援するプロジェクトの例です。

CLAUDE.md（行政書士事務所） 
# 建設業許可申請 書類作成プロジェクト

行政書士事務所の建設業許可申請に必要な書類を作成・管理する
プロジェクト。クライアントごとにフォルダを分けて運用する。

## フォルダ構成

templates/    申請書のひな型（Word・Excel）。ここは読み取り専用。
clients/      顧客別フォルダ。中に申請書類一式を作成する。
reference/    法令資料・記載例・過去の申請事例。参考用。
scripts/      自動化用のPythonスクリプト。
output/       完成した書類の最終版を置く場所。

## よく使う操作

# 新しい顧客フォルダを作るとき
python scripts/new_client.py 会社名

# 書類のWord→PDF変換
python scripts/convert_to_pdf.py ファイル名.docx

# 記載例を確認するとき
reference/ フォルダの中を見る

## ルール

【ファイル命名規則】
  YYYYMMDD_会社名_書類名.拡張子
  例: 20260409_田中建設_許可申請書.docx

【文体】
  申請書類 → 常体（である調）
  クライアントへの案内文 → 敬体（です・ます調）

【テンプレート使用ルール】
  templates/ の中身は絶対に直接編集しない。
  必ず clients/会社名/ にコピーしてから編集すること。

## 注意事項（必ず守ること）

- 代表者の個人番号（マイナンバー）をファイル名に含めない
- 許可番号・登録番号は reference/license_numbers.csv で管理。
  手打ちせず、必ずこのファイルから転記すること
- 提出期限は clients/会社名/deadline.txt に記載がある。
  書類作成時に必ず確認すること
- 完成した書類は必ず output/ にもコピーを置くこと
✅ この例のポイント ① テンプレートを「直接編集するな」と明記している → Claudeが元ファイルを壊す事故を防げる
② 許可番号の参照先を指定している → 手打ちによる転記ミスを防げる
③ ファイル命名規則が具体的 → 「YYYYMMDD」と書くだけでClaude は日付を入れてくれる
④ マイナンバーの注意を書いている → 情報漏洩リスクを構造的に防げる

5. 実例 ② 補助金コンサルの場合

持続化補助金の事業計画書作成を支援するプロジェクトの例です。

CLAUDE.md（補助金コンサル） 
# 持続化補助金 事業計画書作成プロジェクト

小規模事業者持続化補助金の事業計画書を、
クライアントのヒアリング内容をもとに作成するプロジェクト。

## フォルダ構成

hearings/     ヒアリング記録（文字起こし・メモ）
analysis/     市場分析・SWOT分析の結果
drafts/       計画書の下書き（バージョン管理）
final/        提出用の最終版
templates/    様式のひな型。直接編集禁止。
reference/    審査基準・公募要領・記載例

## 作業の流れ

1. hearings/ のヒアリング記録を読む
2. analysis/ に市場分析・SWOT分析を出力
3. drafts/ に事業計画書の各セクションを作成
4. 完成後 final/ に提出用ファイルを出力

## ルール

【文体】
  事業計画書は常体（である調）で統一。
  ただし「お客様の声」の引用部分は敬体でもよい。

【文字数】
  様式2: 経営計画 → 2,000〜3,000字
  様式3: 補助事業計画 → 3,000〜5,000字
  ※ 審査員が読みやすい分量を意識すること。

【数値の扱い】
  売上・経費の数値は必ず hearings/ 内の元データから引用。
  推測で数字を作らない。不明な場合は「要確認」と記載する。

【下書きのバージョン管理】
  drafts/ 内のファイル名: v1_様式名.md, v2_様式名.md ...
  上書きせず、バージョンを増やして保存すること。

## 注意事項

- 数字を「それっぽく」でっち上げない。
  根拠のない数値は審査で減点される。
- クライアントの社名・代表者名が正しいか、
  hearings/ の元データと必ず照合すること。
- 公募要領の締切日は reference/schedule.txt に記載。
  作業前に必ず確認。
- 審査基準の5軸（自社の強み/市場分析/計画の具体性/
  積算の妥当性/補助事業の効果）を必ず網羅すること。
✅ この例のポイント ① 「数字をでっち上げるな」と明記 → AIが最もやりがちなミスを構造的に防止
② 文字数の目安がある → Claudeが長すぎず短すぎない計画書を書ける
③ 審査基準の5軸を書いている → 書き漏らしを防げる
④ 作業の流れが順番に書いてある → Claudeが迷わず段階的に作業できる

6. よくある専門用語をやさしく言い換え

技術者向けの解説でよく出てくる言葉を、実務に置き換えます。

ビルド
書いたものを「完成品」にする作業。たとえばWordの下書きをPDFに変換する、Webサイトのコードを公開用に整えるなど。
テスト
完成品に間違いがないかチェックする作業。書類のダブルチェック、リンク切れの確認など。
リント
文書やコードの「書き方のルール違反」を自動でチェックすること。誤字脱字チェックツールのようなもの。
デプロイ
完成品を「本番の場所」に届ける作業。Webサイトをサーバーに上げる、申請書を電子申請システムに提出するなど。
コマンド
Claude Codeに打ち込む「操作指示」。パソコンに「この作業をやって」と文字で伝えること。
リポジトリ
プロジェクトのファイル一式が入った「仕事用フォルダ」のこと。バージョン管理機能付き。
コミット
作業の「セーブポイント」を作ること。「ここまでの変更を記録に残す」という意味。
git
ファイルの変更履歴を管理するツール。「いつ、誰が、何を変えたか」を自動で記録してくれる日誌のようなもの。
/init
Claude Codeに「このプロジェクトの内容を見て、CLAUDE.mdのたたき台を作って」とお願いする操作。
/clear
Claude Codeの「今日の会話記録」をリセットする操作。CLAUDE.mdの設定は残したまま、頭をスッキリさせる。

7. 最初の一歩 — /init で自動生成する

ゼロから書くのが不安なら、Claudeに下書きを作ってもらえます。

やり方

ターミナル（黒い画面）に打ち込む 
# ① 仕事用フォルダに移動する
cd my-project

# ② Claude Codeを起動する
claude

# ③ 下書きを自動生成する
/init

するとClaudeがフォルダの中身を読み取って、CLAUDE.mdのたたき台を自動で作ってくれます。

ただし、そのまま使わないでください

自動生成されたファイルには「言われなくてもわかること」が大量に含まれます。これを整理するのが大事です。

❌ そのまま使うと…
「このプロジェクトはPythonを使っています」
「ファイルはフォルダに整理されています」
↑ 当たり前すぎて意味がない。
こういう「ノイズ」が多いと、
本当に大事な指示が埋もれます。
✅ 整理すると…
不要な記述を削除して、
自分の事務所固有のルールを追記。
短くて、具体的で、
Claudeが間違えなくなるファイルに。
整理の手順 ① 生成されたファイルを読む
② 「当たり前のこと」を削除する（これが一番大事）
③ 前述の「5つのブロック」に沿って、足りない情報を追記する
④ 使いながら育てる（Claudeが間違えたらその都度ルールを追記）

8. 失敗しないための5つの心がけ

CLAUDE.mdを効果的に使い続けるためのポイント。

❶ 短く書く。目安は印刷して1〜2ページ

CLAUDE.mdは毎回全文読み込まれます。長すぎると、書いてある指示を均等に無視し始めます。「10個のルールを全部守る」より「5個のルールを確実に守る」方が成果が出ます。

❷ やらかしたことをメモする

Claude Codeが間違えたとき、その場で直すだけで終わらせない。CLAUDE.mdに「二度とやるなリスト」として追記する。「テンプレを直接編集した」「存在しない補助金名を書いた」など、一度起きたミスは二度と繰り返さなくなります。

❸ 参照先を具体的に書く

「正確な情報を使うこと」では曖昧すぎます。「reference/license_numbers.csv から転記すること」のように、ファイル名まで指定すると確実です。

❌ 曖昧
正確な情報を使うこと
✅ 具体的
許可番号は reference/license_numbers.csv から転記すること

❹ パスワードや個人情報を書かない

CLAUDE.mdはチームで共有するファイルです。APIキー、パスワード、マイナンバーなどの秘密情報は絶対に書かないでください。個人用のメモは CLAUDE.local.md に。

❺ 月に一度は見直す

仕事の進め方は変わります。古くなったルール、もう使わないフォルダ構成、終わった案件の注意事項…。定期的に掃除して、常に「今の仕事」に合った状態を保ちましょう。

9. 最終チェック

CLAUDE.mdを書き終えたら、この項目を確認してください。

  • ファイル名が CLAUDE.md（大文字）になっている
  • 作業フォルダの一番上（ルート）に置いてある
  • 「このプロジェクトは何か」が1〜3行で書かれている
  • フォルダの役割が説明されている
  • ファイルの命名ルールが具体的に書かれている
  • 「やってはいけないこと」が明記されている
  • 数字や固有名詞の参照先（ファイル名）が具体的に指定されている
  • パスワード・個人情報が含まれていない
  • 印刷したら1〜2ページに収まる長さ

CLAUDE.md は「優秀だけど記憶がない助手」への引き継ぎメモ

最初から完璧を目指さなくて大丈夫です。
まず「プロジェクト概要」と「ファイル命名ルール」と「やってはいけないこと」の3つだけ書く。
あとはClaude Codeが間違えるたびに、1行ずつ追記していけばいい。
使いながら育てる。それが一番確実な方法です。
